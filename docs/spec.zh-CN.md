# Self-Improving AI 规范

本规范描述一套“自我进化”工作流。它不是某个安装器的私有格式，也不要求其他 AI 直接复制 Codex 模板。任何 AI 工具都可以读取本规范，然后按自己的配置机制生成等价实现。

## 核心目标

让 AI 通过可审计的外部脚手架变得更稳定：

```text
观察行为 -> 写候选 -> 用户批准 -> 确定性应用 -> 记录审计日志 -> 用失败案例评测
```

禁止把一次性观察、临时偏好或未经确认的建议静默写入高优先级规则。

## 必须分层

实现时应至少包含这些层：

- 入口规则：工具启动时会读取的稳定说明，例如 `AGENTS.md`、system rules、project instructions。
- 活跃规则：每次都应生效的短规则，例如 `ACTIVE.md`。
- 用户画像：长期偏好，例如 `PROFILE.md`。
- 候选区：自动化和复盘只能写入这里，例如 `CANDIDATES.md`。
- 错误记录：反复失败、排障过程和根因，例如 `ERRORS.md`。
- 经验记录：已验证但还不一定该提升的经验，例如 `LEARNINGS.md`。
- 提升队列：用户批准后的机器可读变更，例如 `promotions/pending/*.yaml`。
- 审计日志：应用、阻止、回滚都要写入日志。
- 评测案例：用 3 到 5 个失败案例检查新规则是否真的改善行为。

工具可以换文件名，但语义边界不能混淆。

## 权限模型

- 自动化可以写候选和报告。
- 自动化不能直接修改入口规则、活跃规则、用户画像、hook、rule 或 skill。
- 只有用户明确批准后，才允许把候选变成 durable rule。
- 批准内容必须机器可读，推荐 YAML 或 JSON。
- 执行器只能做确定性操作，不允许 AI 猜测复杂改动。

推荐 v1 只支持三种操作：

- `append_unique`
- `replace_literal`
- `delete_literal`

无法表达的复杂变更必须进入 `blocked` 并写中文报告。

## 受保护目标

默认只允许提升执行器修改这些高优先级目标：

- 入口规则
- 活跃规则
- 用户画像
- 命令审批规则
- hook 脚本
- self-improvement skill 或等价工作流说明

其他目标默认禁止，除非用户明确扩展白名单。

## 路径和跨平台要求

生成配置时必须注意平台差异：

- Windows 路径不能直接作为 `C:\Users\name\.ai` 写进 JSON/TOML 双引号字符串。
- 推荐在配置文件中使用 `/`：`C:/Users/name/.ai`。
- 如果必须使用反斜杠，必须按目标格式正确转义。
- 写入全局配置前，应先解析原配置；如果原配置不是合法格式，应先备份并停止，不能继续追加。
- 不要把本机绝对路径写进公开模板；模板中使用占位符或由目标 AI 本机生成。

## 用户可读语言

用户可读内容应使用用户偏好的语言。对于中文用户：

- 报告、候选解释、blocked 原因、迁移说明用中文。
- AI 内部 hook 上下文可以用英文，只要更稳定、更少歧义。

## 安全边界

永不记录：

- API key、token、密码、账号凭据
- 私人敏感信息
- session 日志、浏览历史、trusted hook hashes
- backups、本机数据库、临时状态

危险动作应默认阻止或要求确认：

- `rm -rf`
- `git reset --hard`
- `git clean -fdx`
- sudo/admin 权限
- 破坏性系统或磁盘操作

## 给其他 AI 的实现方式

其他 AI 不应盲目复制 Codex 的 `hooks.json` 或 `config.toml`。正确做法是：

1. 阅读本规范。
2. 检查自己的配置机制和权限模型。
3. 生成本工具原生支持的规则、记忆、自动化或工作流。
4. 先 dry-run 或输出计划。
5. 用户确认后再写入本地配置。
6. 写入后运行最小评测。

Codex 目录下的模板只是一个参考实现，不是所有 AI 的标准安装结果。
